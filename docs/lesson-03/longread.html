    🏗 Архитектура IT-продукта

    Из чего состоит IT-продукт?

    Чтобы интегрировать AI в бизнес-системы, нужно понимать, как устроены веб-приложения.

    Четыре слоя любого IT-продукта

    Каждое веб-приложение — от интернет-магазина до CRM — состоит из нескольких слоёв, которые работают вместе.

    👤 Frontend (Фронтенд)
    Интерфейс, кнопки, формы — то, что видит пользователь
    ⇅
    🔌 API (Интерфейс)
    Способ общения между системами — запросы и ответы
    ⇅
    ⚙ Backend (Бэкенд)
    Логика, обработка данных — «мозг» приложения
    ⇅
    🗄 Database (База данных)
    Хранение информации — «память» приложения

    Frontend — что видит пользователь

    Фронтенд отвечает за всё, с чем пользователь взаимодействует в браузере: дизайн, кнопки, формы, анимации.

    🦴

    HTML

    HyperText Markup Language — «скелет» веб-страницы. Определяет структуру: заголовки, текст, картинки, кнопки.

    👗

    CSS

    Cascading Style Sheets — «одежда» страницы. Цвета, шрифты, выравнивание, анимации.

    🧠

    JavaScript

    JS — «нервная система». Интерактивность: кнопки реагируют, всплывающие окна, динамические данные.

    💡 Запомните

    Frontend НИКОГДА не обращается к базе данных напрямую. Всегда через Backend по API. Это фундаментальное правило безопасности.

    ⚙ Backend и базы данных

    Backend и базы данных

    Серверная часть, которая управляет данными и логикой приложения.

    ⚙ Backend — «мозг»

    • Невидимая часть, работает на сервере
    • Обрабатывает данные, проверяет права доступа
    • Языки: Python, Java, C++, PHP, Go
    • Принимает запросы от Frontend, отправляет в БД

    🗄 БД — «память»

    • Организованное хранилище данных
    • Быстрый доступ к информации
    • SQL — язык запросов к базе данных
    • Примеры: PostgreSQL, MySQL, MongoDB
    💡 Пример: личный кабинет курса

    Студент заходит на сайт (Frontend) → Frontend отправляет запрос на Backend → Backend обращается к базе данных → База возвращает данные студента → Backend отдаёт ответ → Frontend показывает профиль.

    🔗 API и JSON

    API и JSON

    Как системы общаются друг с другом.

    📖 API (Application Programming Interface)

    Способ, с помощью которого программы взаимодействуют друг с другом. Одна система отправляет запрос, другая возвращает данные — без необходимости знать внутреннее устройство.

    Как работает API: три шага

    1️⃣

    Запрос

    Программа отправляет запрос: «Дай курс доллара к евро»

    2️⃣

    Обработка

    API передает запрос в систему, которая ищет нужную информацию

    3️⃣

    Ответ

    API возвращает данные: «1 USD = 0.92 EUR»

    📚 Аналогия

    API — это как библиотекарь. Вы спрашиваете: «Можно книгу про AI?» (запрос к API). Библиотекарь ищет книгу в каталоге и приносит её вам (ответ API). Вы можете читать книгу, не зная, как устроен каталог!

    📖 JSON (JavaScript Object Notation)

    Формат хранения и обмена данными между программами. Данные хранятся в виде пар «ключ»: «значение». Это «язык», на котором системы общаются через API.

    { "city": "Moscow", "temperature": -3, "weather": "snowy", "humidity": 85 }

    🌐 HTTP, HTTPS и WebHooks

    HTTP, HTTPS и WebHooks

    Протоколы и способы связи между клиентом и сервером.

    HTTP-методы: три основных

    GET

    Получить данные. Только читает, ничего не изменяет. Примеры: курс валют, список товаров, профиль.

    POST

    Создать / отправить данные. Создаёт новый ресурс. Примеры: оформить заказ, отправить форму.

    DELETE

    Удалить данные. Удаляет ресурс на сервере. Пример: удалить запись из базы данных.

    🔒 HTTPS — безопасный протокол

    HTTPS шифрует все данные между браузером и сервером. Защищает: логины и пароли, платёжные данные, персональную информацию, весь обмен данными.

    ⚡ WebHook — обратный вызов

    ⏳ Обычный API (Polling)

    • Нужно самому отправить запрос
    • Как звонить в пиццерию каждые 5 минут: «Готова ли моя пицца?»
    • Тратит ресурсы на постоянные проверки

    ⚡ WebHook (Push)

    • Сервис сам отправит данные при событии
    • Как пиццерия сама звонит вам, когда пицца готова!
    • Мгновенная доставка без постоянных запросов
    1️⃣ Клиент покупает товар
    →
    2️⃣ Webhook отправляет данные
    →
    3️⃣ Вы видите заказ мгновенно

    A Webhook быстрее, потому что использует HTTP/3

    C Webhook дешевле, потому что не требует сервера
    D Webhook работает без интернета

    No-code vs Разработка

    ✅ No-Low Code

    • ✅ Меньше времени на разработку
    • ✅ Быстрые результаты
    • ✅ Возможности интеграции
    • ❌ Ограниченная производительность на масштабе

    Подходит для: автоматизации, AI-ассистентов, MVP

    💻 Разработка

    • ✅ Гибко настраиваемые модели
    • ❌ Затратно по времени
    • ❌ Дорого, требует разработчиков

    Подходит для: сложных алгоритмов, ML, компьютерного зрения

    ⚠ Проблемы генеративного AI

    Проблемы генеративного AI

    Прежде чем строить AI-агентов, нужно понять ограничения, которые они призваны решить.

    🌫

    Галлюцинации

    Модели выдают правдоподобную, но фактически неверную или вымышленную информацию.

    📅

    Устаревание знаний

    Фиксированная дата обучения — модель не знает о событиях после неё.

    📏

    Ограниченный контекст

    С трудом работают с длинными текстами, нет долгосрочной памяти.

    🔍

    Отсутствие глубины

    Не предоставляют детальную специализированную информацию для сложных запросов.

    ⚠ Пример проблемы

    Запрос: «Что мне делать в оставшиеся дни в Санкт-Петербурге?»
    Модель не знает: сколько дней осталось, какая погода, ваши предпочтения, когда вылет. Результат — общий и неточный ответ.

    💡 Главный вывод

    Именно для решения этих проблем были созданы AI-агенты и RAG-архитектура — о них в следующих разделах.

    🤖 AI агенты

    AI агенты

    Система, способная не только генерировать ответы, но и выполнять сложные действия, взаимодействуя с инструментами и внешними данными.

    Цикл работы AI агента

    AI-агент работает по циклу из четырёх этапов: получает запрос, планирует действия, выполняет их и формирует ответ.

    📩

    Запрос

    Пользователь задаёт вопрос

    🧠

    Plan / Think

    Анализирует задачу, выбирает инструменты

    ⚡

    Act

    Запрашивает данные, использует инструменты

    ✅

    Answer

    Генерирует ответ на основе собранных данных

    🎮 Визуализатор цикла агента — нажимайте «Далее»

    Задача: «Забронируй столик в лучшем итальянском ресторане на пятницу»

    🧩 Проверьте себя

    AI-агент забронировал отель, но перепутал дату. Какой этап цикла сломался?

    A Plan — неправильно спланировал
    B Act — неправильно выполнил бронирование
    C Observe — не проверил результат выполнения
    D Answer — неправильно сформулировал ответ
    📚 RAG

    RAG — Retrieval Augmented Generation

    Генерация с дополнением извлечённой информацией. Многие AI-агенты используют RAG.

    Три этапа RAG

    RAG — гибридный подход, который объединяет генеративные модели с извлечением данных из внешних источников.

    🔍

    1. Извлечение (Retrieval)

    Модель ищет релевантные документы из базы данных или корпуса с помощью алгоритмов поиска.

    ➕

    2. Дополнение (Augmentation)

    Извлечённые данные структурируются и дополняют исходный запрос контекстом для точного ответа.

    ✨

    3. Генерация (Generation)

    Модель создаёт ответ на основе запроса и извлечённой информации, обеспечивая фактическую точность.

    Преимущества RAG

    ✅ Уменьшение галлюцинаций

    Фактическая и актуальная информация из реальных источников.

    🔄 Актуальность данных

    Доступ к обновляемым базам и документам.

    📏 Расширение контекста

    Увеличение «контекстного окна», связность при длительных взаимодействиях.

    📊 Углубление ответа

    Детализация и специфичность за счёт извлечённых данных.

    🧩 Проверьте себя

    Зачем в RAG нужен этап Augmentation, если можно сразу отдать документы модели?

    A Augmentation не нужен — это лишний шаг
    B Нужно структурировать контекст и правильно сформулировать задачу для модели
    C Augmentation переводит документы на другой язык
    D Augmentation сжимает документы для экономии токенов
    🌍 Экосистема AI

    Экосистема AI и диапазон сложности

    Какие модели и платформы доступны для создания AI-систем.

    ☁ Проприетарные (Closed-source)

    • OpenAI, Google, Anthropic — через облачные API
    • 💳 Оплата: за токены (вход + выход)
    • ✅ Быстрый старт, высокая производительность
    • ❌ Зависимость от провайдера, стоимость растёт

    💻 Open-source

    • LLaMA, Mistral, Falcon, Qwen — запуск локально
    • 💻 Оплата: за ресурсы (GPU, CPU, RAM)
    • ✅ Полный контроль, кастомизация
    • ❌ Нужны DevOps-знания, затраты на железо

    Диапазон сложности AI-систем

    LLM (без агента)

    Изолированно, нет инструментов. Отвечает на основе обученных знаний.

    LLM воркфлоу

    Подключены инструменты и БД через API. Последовательность действий.

    AI агент

    Сам выбирает инструменты, многошаговые цепочки.

    Мультиагенты

    Команда агентов: один планирует, другой ищет, третий анализирует.
