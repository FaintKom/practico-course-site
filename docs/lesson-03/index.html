Из чего состоит IT-продукт?

Как интегрировать AI в ваши системы? Для этого нужно понимать основы веб-разработки. Давайте разбираться.

Интерфейс, кнопки, формы — то, что видит пользователь
⇅

Способ общения между системами — запросы и ответы
⇅

Логика, обработка данных — «мозг» приложения
⇅

Хранение информации — «память» приложения

Frontend — что видит пользователь

Фронтенд — часть веб-разработки, которая отвечает за то, что пользователь видит и с чем взаимодействует в браузере: интерфейс, дизайн, кнопки, формы, анимации.

🦴
HTML
HyperText Markup Language — «скелет» веб-страницы. Отвечает за структуру: текст, заголовки, картинки, кнопки, таблицы.
👗
CSS
Cascading Style Sheets — «одежда» для страницы. Добавляет цвета, меняет шрифты, выравнивает элементы, делает анимации.
🧠
JavaScript
JS — «нервная система» страницы. Добавляет интерактивность: кнопки реагируют на нажатие, всплывающие окна, автозаполнение.

Backend и базы данных

Невидимая часть IT-продукта, которая управляет данными и логикой.

⚙️ Backend — «мозг» приложения

Невидимая часть сайта или приложения, которая работает на сервере. Отвечает за обработку данных, логику работы и взаимодействие с базами данных.

Python Java C++ PHP + другие

🗄️ База данных — «память» приложения

Организованное хранилище данных для хранения, управления и быстрого доступа к информации.

Пример: личный кабинет курса. Фронтенд показывает страницу, бэкенд отправляет запрос в базу данных, база возвращает данные пользователя.

API и JSON

Как программы общаются друг с другом и обмениваются данными.

🔌 API — интерфейс для соединения сервисов

API (Application Programming Interface) — способ, с помощью которого программы взаимодействуют друг с другом. Позволяет одной системе отправлять запросы другой и получать данные без необходимости знать, как та работает внутри.

1️⃣
Запрос
Программа отправляет запрос (например: «Дай курс доллара к евро»)
2️⃣
Обработка
API передает запрос в систему, которая ищет нужную информацию
3️⃣
Ответ
API возвращает данные (например: «1 USD = 0.92 EUR»)

📚 Аналогия: API — это как библиотекарь. Вы спрашиваете: «Можно мне книгу про AI?» (запрос к API). Библиотекарь ищет книгу в каталоге и приносит ее вам (ответ API). Вы можете читать книгу, не зная, как устроен каталог!

📦 JSON — язык общения сервисов

JSON (JavaScript Object Notation) — формат хранения данных, который помогает программам обмениваться информацией.

📩 Запрос к API:

«Скажи мне погоду в Берлине?»

📨 Ответ от API (JSON):

// Ответ API погоды { "city": "Berlin", "temperature": +5, "unit": "Celsius", "weather": "cloudy" }

🌤️ Наше приложение читает JSON и показывает: «В Берлине сейчас +5°C, облачно.»

ПРОТОКОЛЫ

HTTP, HTTPS и WebHooks

Способы связи между клиентом и сервером.

GET
Получить данные. Только читает, ничего не изменяет. Примеры: получить курс валют, список товаров, профиль пользователя.
POST
Создать / отправить данные. Создает новый ресурс или запускает действие. Примеры: оформить заказ, отправить форму.
DELETE
Удалить данные. Удаляет ресурс на сервере. Пример: удалить запись из базы данных.

🔐 HTTPS защищает:

✅ Логины и пароли ✅ Платежные данные ✅ Персональные данные ✅ Весь обмен данными

⚡ WebHook — обратный вызов

⏳ Обычный API

Нужно самому отправить запрос, чтобы получить данные. Как если бы вы звонили в пиццерию каждые 5 минут: «Готова ли моя пицца?»

⚡ Webhook

Сервис сам отправит данные, как только что-то произошло. Как если бы пиццерия сама позвонила вам, когда пицца готова!

Пример: интернет-магазин с Webhook

1️⃣ Клиент покупает товар
→
2️⃣ Webhook отправляет данные
→
3️⃣ Вы видите заказ мгновенно

🍕 Аналогия: Заказ пиццы

API — звоним в пиццерию: «Какие пиццы есть?»
Webhook — пиццерия сама звонит, когда пицца готова
HTTPS — безопасный канал связи (телефон)
JSON — язык меню: { "Маргарита": 10€ }

📝 Проверьте себя: Чем Webhook отличается от обычного API?

Webhook работает быстрее
Webhook — «обратный вызов»: сервер сам отправляет данные при наступлении события
Webhook используется только для безопасности
Webhook не поддерживает JSON
Webhook — push-модель: сервер автоматически отправляет данные на указанный URL при наступлении события, а не клиент постоянно спрашивает.

No-code vs Разработка

✅ No-Low Code

  • ✅ Меньше времени на разработку
  • ✅ Быстрые результаты
  • ✅ Возможности интеграции
  • ❌ Ограниченная производительность на большом масштабе

Подходит для: автоматизации процессов, AI-ассистентов, MVP-продуктов

💻 Разработка

  • ✅ Гибко настраиваемые модели
  • ❌ Затратно по времени
  • ❌ Дорого, требует разработчиков

Подходит для: сложных алгоритмов, предсказательных моделей, компьютерного зрения, ML

ПРОБЛЕМЫ

Проблемы генеративного AI

Основные ограничения, с которыми сталкиваются LLM-модели.

🌫️
Галлюцинации
Модели могут выдавать правдоподобную, но фактически неверную или вымышленную информацию.
📅
Устаревание знаний
Модели имеют фиксированную дату обучения — неподходящи для задач, требующих актуальной информации.
📏
Ограниченный контекст
С трудом работают с длинными текстами или задачами, требующими долгосрочной памяти.
🔍
Отсутствие глубины
Могут не предоставлять детальную и специализированную информацию для сложных запросов.

Пример проблемы:

Запрос: «Что мне делать в оставшиеся дни в Санкт-Петербурге?»

Модель не знает: сколько дней осталось, какая погода, ваши предпочтения, когда вылет. Результат будет общим и неточным.

AI агенты

Система на основе генеративных моделей AI, способная не только генерировать ответы, но и выполнять сложные действия, взаимодействуя с инструментами и внешними данными.

АРХИТЕКТУРА

RAG — Retrieval Augmented Generation

Генерация с дополнением извлеченной информацией. Многие AI-агенты используют RAG.

RAG — гибридный подход, который объединяет генеративные модели с методами извлечения данных из внешних источников.

🔍
1. Извлечение (Retrieval)
Модель ищет и извлекает релевантные документы из базы данных или корпуса. Используются алгоритмы или нейронные модели для нахождения подходящих фрагментов.
➕
2. Дополнение (Augmentation)
Извлеченные данные дополняют исходный запрос, обеспечивая модель дополнительным контекстом для более точного ответа.
✨
3. Генерация (Generation)
Модель создает ответ на основе оригинального запроса и извлеченной информации, обеспечивая фактическую обоснованность.

Кейс: HR-агентство с RAG

📩 Запрос: «Найди Senior Python-разработчика»
→
🤖 LLM анализирует запрос
→
🗄️ Поиск в базе кандидатов
→
✅ Ответ с релевантными кандидатами

Преимущества RAG

Уменьшение галлюцинаций
Извлечение данных позволяет использовать фактическую и актуальную информацию.
Актуальность данных
Доступ к обновляемым базам и открытой информации.
Расширение контекста
Увеличение «контекстного окна», связность при длительных взаимодействиях.
Углубление ответа
Извлеченные данные добавляют детализацию и специфичность.
Эффективность
Сужение области поиска помогает модели сосредоточиться на релевантной информации.
Применение в бизнесе
Q&A системы, генерация контента, клиентская поддержка из баз знаний.

ЭКОСИСТЕМА

Экосистема AI и диапазон сложности

На базе чего создавать агентские системы? Проприетарные vs Open-source модели.

1. Проприетарные модели (Closed-source)

OpenAI, Google, Claude — доступ только через облачные API.

  • 💳 Оплата: за токены (вход + выход)
  • ✅ Быстрый старт, высокая производительность
  • ✅ Доступ к свежим версиям моделей
  • ❌ Зависимость от провайдера
  • ❌ Стоимость растет при объемах

2. Open-source модели

LLaMA, Mistral, Falcon, Mixtral, Qwen — можно запускать локально.

  • 💻 Оплата: за ресурсы (GPU, CPU, RAM)
  • ✅ Полный контроль над моделью
  • ✅ Кастомизация и дообучение
  • ❌ Нужны DevOps-знания
  • ❌ Затраты на железо

Диапазон сложности AI-систем

LLM (без агента)

Работает изолированно, нет инструментов, памяти и доступа к данным. Отвечает на основе обученных знаний.

LLM воркфлоу

Подключаются инструменты и базы данных через API. Модель выполняет последовательность действий.

AI агент

Сам выбирает инструменты, строит многошаговые цепочки, улучшает свои промпты.

Мультиагентные системы

Команда агентов с распределенными ролями: один планирует, другой ищет, третий анализирует.

Создание кастомного GPT-агента

Пошаговая инструкция по созданию своего AI-агента в OpenAI Platform.

0

Регистрация в OpenAI Platform

Заходим на platform.openai.com. Если нет аккаунта — регистрируемся. Для работы необходим VPN и иностранный номер. Нажимаем Sign up и создаем учетную запись через Google почту.

1

Start Building

Нажимаем на верхней панели кнопку «Start Building» для начала создания проекта.

2

Создание организации

Заполняем поля: Organization name и «What best describes you» — выбираем «Not technical». Нажимаем «Create organization».

3

Настройка API ключей

Переходим в раздел API Keys. Создаем новый секретный ключ — он понадобится для интеграций.

4

Создание Assistant

Переходим в раздел Assistants. Нажимаем «Create» и задаем имя, инструкции и выбираем модель (GPT-4o рекомендуется).

5

Настройка инструкций

Пишем системный промпт: описываем роль, контекст, ограничения и стиль ответа агента. Это основа качества работы!

6

Загрузка файлов (Knowledge Base)

Загружаем документы компании — FAQ, инструкции, каталоги. Агент будет использовать их через RAG для точных ответов.

7

Настройка параметров

Настраиваем Temperature (креативность), Max tokens (длина ответа), Top-p и другие параметры модели.

8

Подключение инструментов

Включаем Code Interpreter для анализа данных, File Search для поиска по документам, Function Calling для интеграций.

9

Тестирование

Используем Playground для тестирования агента. Проверяем качество ответов, точность, использование загруженных документов.

💡 Ключевые советы

  • ✅ Чем подробнее системный промпт — тем лучше результат
  • ✅ Загружайте актуальные документы для RAG
  • ✅ Тестируйте разные значения Temperature (0.0-1.0)
  • ✅ Используйте Function Calling для интеграций с внешними API
